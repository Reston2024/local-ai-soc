<objective>
Move Ollama inference from CPU to the RTX 5080 (16 GB VRAM, CUDA 13.1). On Windows, Ollama runs as a system service and does not inherit user-session environment variables. The fix is to set `CUDA_VISIBLE_DEVICES=0` at the Machine (system) scope, then restart the Ollama service so it picks up the new env var. This is a manual, privileged operation — executor must run PowerShell as Administrator. The GPU migration unblocks all subsequent phases that depend on reasonable TTFT for LLM inference (target: under 30 seconds vs ~300 seconds CPU).
</objective>

<task type="manual">
  <name>Task 1: Set CUDA_VISIBLE_DEVICES at Machine scope (run as Administrator)</name>
  <files>Windows System Environment Variables</files>
  <action>
Open PowerShell as Administrator.

Check current system-level value:
```powershell
[System.Environment]::GetEnvironmentVariable("CUDA_VISIBLE_DEVICES", "Machine")
```
Note the output (likely null/empty).

Set the system env var:
```powershell
[System.Environment]::SetEnvironmentVariable("CUDA_VISIBLE_DEVICES", "0", "Machine")
```

DO NOT set OLLAMA_HOST at Machine scope — this conflicts with the normalize_ollama_host validator in backend/core/config.py which handles 0.0.0.0 conversion correctly for the Python client.

Verify the value was written:
```powershell
[System.Environment]::GetEnvironmentVariable("CUDA_VISIBLE_DEVICES","Machine")
```
Should return `"0"`.
  </action>
  <verify>
    <automated>powershell -Command "[System.Environment]::GetEnvironmentVariable('CUDA_VISIBLE_DEVICES','Machine')"</automated>
  </verify>
  <done>[System.Environment]::GetEnvironmentVariable("CUDA_VISIBLE_DEVICES","Machine") returns the string "0".</done>
</task>

<task type="manual">
  <name>Task 2: Restart the Ollama Windows service</name>
  <files>Ollama Windows service</files>
  <action>
In the Administrator PowerShell session:
```powershell
Stop-Service -Name "Ollama" -Force
Start-Service -Name "Ollama"
Get-Service -Name "Ollama" | Select-Object Status
```

If Stop-Service/Start-Service fail because the service name differs, find the correct name:
```powershell
Get-Service | Where-Object {$_.Name -like "*ollama*" -or $_.DisplayName -like "*ollama*"} | Select-Object Name,DisplayName
```

Then use the correct name in Stop-Service/Start-Service.
  </action>
  <verify>
    <automated>powershell -Command "Get-Service -Name 'Ollama' | Select-Object -ExpandProperty Status"</automated>
  </verify>
  <done>Get-Service -Name "Ollama" shows Status = Running after restart.</done>
</task>

<task type="manual">
  <name>Task 3: Verify GPU layers during active inference</name>
  <files>N/A — verification only</files>
  <action>
Open two PowerShell windows side by side.

In window 1, start a generation:
```
ollama run qwen3:14b "Summarize MITRE ATT&CK T1059 in one sentence"
```

Immediately in window 2, while generation is running:
```
ollama ps
```

Look for the "GPU Layers" column — value must be > 0 (typically 40+ for qwen3:14b on 16 GB VRAM).

Also verify GPU utilization:
```
nvidia-smi dmon -s u -d 1 -c 5
```
Should show nonzero GPU utilization during inference.

If GPU Layers = 0 after service restart, the service may not have picked up the env var. In that case, reinstall Ollama while the system env vars are set:
```
winget uninstall Ollama.Ollama
winget install Ollama.Ollama
```
The installer registers a new service that reads system env at install time.

Time a generation to confirm speedup:
```powershell
$t = Get-Date; ollama run qwen3:14b "What is T1059?" ; (Get-Date) - $t | Select-Object TotalSeconds
```
TTFT should be under 30 seconds on GPU. Previously on CPU this was ~300 seconds.
  </action>
  <verify>
    <automated>ollama ps</automated>
  </verify>
  <done>ollama ps shows GPU Layers > 0 while qwen3:14b is loaded. nvidia-smi shows GPU utilization > 0%. Generation completes in under 30 seconds for a short prompt.</done>
</task>
