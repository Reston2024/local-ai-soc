## Goal

Add a heuristic 0.0–1.0 confidence score computed after every LLM generation: store it in a new `confidence_score` column on `llm_audit_provenance`, and surface it as a non-dismissable confidence badge in InvestigationView's copilot panel.

## Context

**llm_audit_provenance DDL (sqlite_store.py lines 229–239):**
```sql
CREATE TABLE IF NOT EXISTS llm_audit_provenance (
    audit_id                TEXT PRIMARY KEY,
    model_id                TEXT NOT NULL,
    prompt_template_name    TEXT,
    prompt_template_sha256  TEXT,
    response_sha256         TEXT,
    operator_id             TEXT,
    grounding_event_ids     TEXT NOT NULL DEFAULT '[]',
    created_at              TEXT NOT NULL
);
```
No `confidence_score` column yet.

**Pitfall:** Add `confidence_score REAL` to BOTH the `_DDL` string (for fresh databases) AND as an `ALTER TABLE` in `__init__` (for existing `data/graph.db`). If only ALTER TABLE, in-memory test databases won't have the column.

**Heuristic algorithm (from research Pattern 5):**
```python
score = 0.0
if grounding_event_ids is not None and len(grounding_event_ids) > 0:
    score += 0.5   # has grounding context
if citation_verified is True:
    score += 0.3   # all citations resolved
if len(grounding_event_ids or []) >= 5:
    score += 0.1   # rich context
if prompt_template_sha256 is not None:
    score += 0.1   # known template
score = min(score, 1.0)
```
This is computed in `query.py` using values already available: `ids` (from Chroma), `citation_ok` (from `verify_citations()`), and `out_ctx` (populated after P22-T01 changes). Note: Plan 22-02 is Wave 1 parallel to 22-01. The `out_context` pattern added in 22-01 is needed for `update_confidence_score` call. If 22-02 executes before 22-01 is merged, implement the score computation in `query.py` but stub the `update_confidence_score` call behind a guard: `if audit_id := out_ctx.get("audit_id"):`. The heuristic computation itself does not depend on 22-01 changes.

**UI badge thresholds (ROADMAP P22-T02 says green ≥0.8, amber 0.5–0.8, red <0.5; research says green ≥0.7, amber 0.3–0.7, red <0.3 — use ROADMAP values as the requirement source):**
- `high`: score >= 0.8 → green badge
- `medium`: 0.5 <= score < 0.8 → amber badge
- `low`: score < 0.5 → red badge

**ChatHistoryMessage in api.ts** currently has: `id`, `investigation_id`, `role`, `content`, `created_at`. Add `confidence?: number` and `audit_id?: string` as optional fields — backward compatible.

**InvestigationView.svelte** copilot panel renders chat messages. Existing pattern for conditional class binding: `class="timeline-entry severity-{item.severity ?? 'info'}"`. Apply same pattern for confidence badge.

```python
# Heuristic confidence score (P22-T02)
_score = 0.0
if ids:
    _score += 0.5
if citation_ok:
    _score += 0.3
if len(ids) >= 5:
    _score += 0.1
# prompt_template_sha256 not passed in ask() — use None check below
_score = min(_score, 1.0)
confidence_score = round(_score, 4)

# Persist score to provenance record (non-fatal)
_audit_id = out_ctx.get("audit_id") if "out_ctx" in dir() else None
if _audit_id and stores.sqlite is not None:
    import asyncio as _asyncio
    try:
        await _asyncio.to_thread(
            stores.sqlite.update_confidence_score,
            _audit_id,
            confidence_score,
        )
    except Exception as exc:
        log.warning("Confidence score write failed (non-fatal)", error=str(exc))
```
