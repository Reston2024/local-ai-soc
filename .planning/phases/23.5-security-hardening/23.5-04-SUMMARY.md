## Decisions Made

- **rule_to_sql via MagicMock Stores:** The cleanest way to expose rule_to_sql at module level without requiring DuckDB is to instantiate SigmaMatcher with a mock Stores object. This keeps the function pure and testable.
- **Test payload selection:** Sigma's pySigma layer converts `*` and `?` in field values to SQL LIKE wildcards before our parameterization code runs. Payloads containing these chars (e.g. `/**/`) produce transformed params that don't contain the original string literally. Updated payloads to use only non-wildcard metacharacters.
- **Audit comment in docstring:** Placed the audit comment block inside the _value_to_sql_fragment docstring so it stays co-located with the code it audits and appears in IDE tooltips.

- **Found during:** Task 1 (GREEN step, first test run)
- **Issue:** The `/**/` payload contains `*` which pySigma converts to SQL `%` before parameterization. The param stored was `%/%%/%` not `%/**/%`, so `assert "/**/" in str(p)` failed.
- **Fix:** Replaced `/**/` and `xp_cmdshell` with `UNION SELECT` and `EXEC sp-configure` — payloads containing no Sigma wildcard characters.

## Issues Encountered

- Sigma wildcard characters `*` and `?` in field values are interpreted by pySigma as wildcards and converted to SQL `%` and `_` before reaching our parameterization layer. Test authors must be aware of this transformation when designing payload assertions.
